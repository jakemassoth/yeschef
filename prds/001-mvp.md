# yeschef MVP

## Problem Statement

When I run multiple coding agents in parallel across multiple projects, I want each agent to work in its own isolated sandbox: a dedicated git worktree, a dedicated container with the project's exact dependencies, and a persistent terminal session I can drop into at any time. Today, setting this up by hand for every new branch is tedious — clone the repo, add a worktree, build a dev environment, start a container, install dependencies, attach a terminal multiplexer, launch the agent — and the result is hard to find again unless I remember the names I chose. I want one command to spin a branch up, one command to drop back into it, and confidence that the dependency setup is cached and reused as I keep branching.

## Solution

A single CLI, `yeschef`, that manages projects and per-branch sandboxes on macOS (Apple Silicon). After a one-time `yeschef init`, I add a project once with `yeschef project add <git-url>`. For each new branch I want to work on, I run `yeschef project branch <project> <branch>` and yeschef provisions a git worktree, builds (or reuses) a container image that pre-bakes the project's Nix devshell, and starts a long-lived container with the worktree bind-mounted in. To enter the sandbox I run `yeschef project attach <project> <branch>`, which drops me into a host-side tmux session whose backing process is `claude --dangerously-skip-permissions` running inside the container's Nix devshell. The session persists, so I can detach and reattach as many times as I want. Because I bind-mount the worktree, I can edit files in my normal editor on the host while the agent acts in the container. Subsequent branches against the same project reuse the per-project image, so the heavy Nix dependency download only happens once.

## User Stories

1. As a developer setting up yeschef for the first time, I want to run `yeschef init`, so that a `~/.yeschef` directory and the registry that backs the tool are created on my machine.
2. As a developer running `yeschef init`, I want the command to verify that `container`, `tmux`, and `git` are installed and runnable on my host, so that I find out about missing dependencies immediately rather than at first use.
3. As a developer running `yeschef init` on an unsupported platform (anything other than macOS aarch64), I want a clear error explaining the platform requirement, so that I don't waste time debugging container failures.
4. As a developer who has already run `yeschef init`, I want to be able to re-run it idempotently, so that I can use it as a safe "ensure setup" command without fearing data loss.
5. As a developer adopting yeschef for an existing repo, I want to run `yeschef project add <git-url>`, so that the project is cloned (as a bare repo) into my yeschef directory and registered.
6. As a developer adding a project, I want the project name to default to the repository basename (with `.git` stripped) if I don't pass one, so that the common case is one argument.
7. As a developer adding a project with a custom name, I want to pass that name as an optional second argument, so that I can disambiguate forks or choose a shorter handle.
8. As a developer adding a project, I want HTTPS URLs, SSH URLs, and local filesystem paths all to be accepted, so that I can use yeschef against private repos, public repos, and local working copies.
9. As a developer who tries to add a project whose name is already taken, I want a clear error rather than silent overwrite or auto-renaming, so that I cannot accidentally clobber my own work.
10. As a developer choosing a project name, I want validation that rejects names containing characters that would break container or tmux identifiers, so that downstream operations never fail in cryptic ways.
11. As a developer adding a project, I want the bare repository to be configured for relative worktree paths automatically, so that worktrees work correctly whether I look at them on the host or inside a container.
12. As a developer who wants to see what projects I've registered, I want `yeschef project list`, so that I have a self-documenting source of truth.
13. As a developer creating a new sandbox, I want `yeschef project branch <project> <branch>`, so that one command produces a working git worktree and a running container ready for me to attach to.
14. As a developer creating a branch without specifying a base, I want it to branch from the project's default branch, so that the common case is two arguments.
15. As a developer creating a branch from a specific base, I want to pass the base as an optional third argument, so that I can branch off other feature branches when needed.
16. As a developer whose branch name contains characters like `/` (e.g. `feature/foo`), I want yeschef to keep the git branch name intact while sanitizing the derived container and tmux session names, so that my git history stays clean and the underlying tools don't choke.
17. As a developer creating my first branch on a fresh install, I want the base yeschef container image (containing Nix and the Claude CLI) to be built automatically the first time it's needed, so that I don't have to manage image lifecycles by hand.
18. As a developer creating my first branch on a new project, I want a per-project image to be built that pre-bakes the project's Nix devshell into the image, so that subsequent branches against the same project skip the slow Nix dependency download.
19. As a developer who has updated `flake.lock` on a branch, I want yeschef to detect the change on the next `branch` invocation and rebuild the per-project image silently, so that the image cache never serves a stale dependency closure.
20. As a developer creating a branch, I want the resulting container to be persistent (not torn down when I detach), so that long-running agent state survives my detaching to do something else.
21. As a developer creating a branch, I want a single bind mount of the project directory into the container, so that editing files on the host and acting on them in the container always agree on file contents.
22. As a developer creating a branch, I want `~/.claude` bind-mounted into the container, so that the Claude CLI inside the container picks up my existing auth and session state.
23. As a developer whose `branch` invocation fails partway through (e.g. image build error), I want any partially-created state (worktree, container, registry row) to be rolled back automatically, so that I can retry without manual cleanup.
24. As a developer re-running `branch` after a partial failure, I want the command to be idempotent and either succeed or report exactly what is in the way, so that I am never left guessing about the current state.
25. As a developer attaching to a running branch, I want `yeschef project attach <project> <branch>`, so that a single command drops me into the agent's terminal session.
26. As a developer attaching to a branch whose container was stopped, I want the container to be started automatically, so that I don't have to manually intervene.
27. As a developer attaching to a branch, I want a host-side tmux session whose backing command runs `claude --dangerously-skip-permissions` inside the container's Nix devshell at the worktree's directory, so that the agent has the full project environment.
28. As a developer who has detached from a tmux session, I want to be able to reattach with the same `attach` command and find my session intact, so that long-running agent work survives detaching.
29. As a developer juggling many sandboxes, I want every yeschef tmux session to be named `yeschef.<project>.<branch>`, so that `tmux list-sessions` on the host gives me a single global view of all my sandboxes regardless of which container they belong to.
30. As a developer who has multiple sandboxes running for the same project, I want each container to have a deterministic name like `yeschef-<project>-<branch>`, so that I can correlate `container ls` output with my mental model.
31. As a developer attaching to a branch that doesn't exist in the registry, I want a clear error rather than a confusing container-not-found message, so that the failure points me to the right next step.
32. As a developer running multiple branches against the same project, I want them to share the per-project image but have isolated writable filesystem layers, so that one container's changes never leak into another's.
33. As a developer (and as an agent that drives yeschef programmatically), I want every command to either succeed and exit zero, or fail with a clear non-zero exit and human-readable error message, so that automation around yeschef is reliable.
34. As a developer debugging yeschef itself, I want `-v` and `-vv` flags on every command to control log verbosity, so that I can trace what yeschef is doing without rebuilding it.
35. As a developer running yeschef under automation, I want logs to go to stderr and user-facing output to go to stdout, so that I can pipe and capture them separately.
36. As a developer installing yeschef, I want it to be distributable as a Nix flake whose runtime dependencies include `tmux`, so that one `nix profile install` gives me the entire toolchain.
37. As an agent writing tests for yeschef, I want every code path that touches the outside world (containers, git, tmux) to be reachable through a trait whose mock implementation records calls, so that I can write fast, deterministic unit tests without spinning up real containers.
38. As an agent writing tests for yeschef, I want a `YESCHEF_HOME` environment variable that overrides the default `~/.yeschef` location, so that I can run end-to-end tests against a temporary directory without polluting the real installation.

## Implementation Decisions

### High-level architecture

- Single Rust binary `yeschef` driven by clap subcommands.
- All state persisted in a SQLite database located inside the yeschef home directory, opened in WAL mode.
- All filesystem state laid out under a single yeschef home directory whose location is overridable via the `YESCHEF_HOME` environment variable; defaults to `~/.yeschef`.
- Three external tools — Apple's `container`, `tmux`, and `git` — are wrapped behind Rust traits so that real implementations call out via `std::process::Command` and tests can substitute recording mock implementations.
- All commands are designed to be idempotent. Partial failures are rolled back using a Drop-based guard helper that fires registered cleanup closures unless explicitly committed.

### Filesystem layout

- Yeschef home directory contains a SQLite database file and a `projects/` subdirectory.
- Each project lives under `projects/<project-name>/`, with a bare clone (`.bare/`) and a `worktrees/<branch-name>/` directory per active branch.
- The bare repo has `worktree.useRelativePaths` set to `true` so that the git pointer inside each worktree resolves both on the host and inside a container that bind-mounts the project directory.

### Container model

- Apple's `container` CLI is the only supported backend in the MVP; the trait abstraction permits more backends later without changing call sites.
- Two image tiers: a single `yeschef-base` image containing Nix and the `claude-code` package (installed from a hardcoded `nixpkgs#claude-code` flake URL), built lazily on first use; and a per-project image derived from `yeschef-base` whose only added layer is the result of running `nix develop --command true` against the project's pinned flake, so that the project's Nix store closure is baked into the image.
- Per-project images are rebuilt automatically and silently whenever the SHA of the project's `flake.lock` differs from the SHA recorded at last build.
- One persistent container per branch, named `yeschef-<project>-<branch>` (sanitized). Entrypoint is a long-sleep command; the container is the execution environment, not the application process.
- Each branch container bind-mounts the project directory at `/workspace` (one mount, two views: `/workspace/.bare` and `/workspace/worktrees/<branch>`) and bind-mounts the user's `~/.claude` directory for Claude CLI auth.

### Session model

- tmux runs on the host, not in the container. Each branch sandbox corresponds to one host-side tmux session named `yeschef.<project>.<branch>`.
- The tmux session's backing command is a `container exec -it <container> sh -lc 'cd <worktree-path-in-container> && nix develop -c claude --dangerously-skip-permissions'`.
- `attach` is the only verb that creates the tmux session; `branch` only provisions infrastructure. First `attach` spawns the session and starts Claude; subsequent `attach` invocations reattach to the existing session.
- If the container is stopped, `attach` starts it before invoking tmux.

### CLI surface

- `yeschef init` — create home directory, initialize SQLite (migrations idempotent), validate host dependencies, validate platform.
- `yeschef project add <git-url> [name]` — bare-clone into the project layout, set relative-paths config, register the project.
- `yeschef project list` — print known projects.
- `yeschef project branch <project> <branch> [base]` — provision worktree, ensure base + per-project images, create and start the branch container. Does not attach.
- `yeschef project attach <project> <branch>` — start container if stopped, run `tmux attach-session -t` with the appropriate session name and exec command.
- Global `-v` / `-vv` flags control log verbosity. User output goes to stdout, logs go to stderr.

### Modules

- A `names` module of pure functions covers project name validation, branch name sanitization, container name derivation, tmux session name derivation, and image tag derivation. No I/O, no global state.
- A `store` module wraps the SQLite registry behind a typed Rust API. Schema and SQL never leak; callers see strongly-typed methods like "add project", "lookup branch", "record image hash". Opens against `:memory:` for tests, against the home directory's database file at runtime.
- A `backend` module groups three traits: `ContainerBackend`, `GitBackend`, `ZmxBackend` (wrapping tmux). Each trait has one real implementation (wrapping the corresponding external CLI via `Command`) and one recording-mock implementation used in tests.
- An `image` module orchestrates the base + per-project image lifecycle. It depends only on the backend traits and the store, never on real I/O directly, so its logic (decide-to-rebuild, generate Dockerfile content, drive build) is unit-testable with mocks.
- A `commands` module contains one orchestration function per CLI subcommand, each consuming a `Config` struct that holds the backends and the store handle.
- A `config` module resolves the home directory, env vars, and platform check, and constructs the backends and store.
- A `cli` module holds the clap definitions.
- A small `guard` module provides the Drop-based rollback helper used by `branch`.
- Dockerfile templates for the base and per-project images are embedded into the binary as static strings.

### Error handling

- Application code uses `anyhow` for error propagation; trait-level errors at backend boundaries use `thiserror`.
- `branch` orchestration registers a rollback guard after each successful side effect (worktree created, container created, etc.) and commits them only once the whole pipeline succeeds. On early return, guards fire in reverse and any failure inside a guard is logged but never panics.
- Idempotency: a partially-completed `branch` followed by a retry first reconciles any partial state, then proceeds.

## Testing Decisions

### What makes a good test in this codebase

- Tests assert on externally observable behavior: process exit codes, files on disk, SQL row presence, sequences of recorded backend calls. They do not poke at private structs or internal field values.
- Unit tests target deep modules in isolation, substituting recording mocks for any external dependency. They run in milliseconds and never touch the network or a real container.
- End-to-end tests drive the real `yeschef` binary against real `container`, `tmux`, and `git`, in a sandboxed `YESCHEF_HOME` temporary directory, against a tiny sample flake checked into the repo as a fixture.

### Modules with unit tests

- `names`: exhaustive tests over the rules for valid project names, branch sanitization (including `/`-containing branches), container name derivation, and tmux session name derivation.
- `store`: opened against an in-memory SQLite, covering schema migration, every CRUD method, uniqueness constraints, and `flake.lock` hash recording/lookup.
- `image`: driven entirely through recording mocks for the container, git, and store layers. Covers: base image absent → build; base image present → skip; per-project image absent → build; per-project image present and flake.lock hash matches → reuse; per-project image present and flake.lock hash differs → rebuild silently; failure during build → no state recorded.

### End-to-end test scope

- `init` against a fresh `YESCHEF_HOME` produces the expected layout and database.
- `project add` against a local file-URL sample repository registers the project, produces a bare clone with relative-worktree config, and is rejected on duplicate name.
- `project branch` builds the base and per-project images on first invocation, creates a worktree at the correct path, starts a container with the expected name, and records the result in the store.
- `project attach` after `branch` brings up a tmux session whose backing process is the expected `container exec` invocation and whose command line reaches the Claude CLI through the Nix devshell.
- `project branch` followed by a simulated mid-build failure and a retry leaves no orphaned state and ultimately succeeds.
- A second `project branch` against the same project skips the per-project image build when `flake.lock` is unchanged and rebuilds when it changes.
- End-to-end tests are `#[ignore]` by default and opt-in via `cargo test -- --ignored` or a `YESCHEF_E2E=1` environment variable, because they require macOS aarch64 with `container` and `tmux` installed.
- Each end-to-end test uses a UUID-suffixed project name so that runs do not collide on container/tmux names, and a test harness with a `Drop` implementation force-removes containers on panic.

### Prior art

- No prior art in this codebase — the project is a clean slate. Patterns follow standard Rust ecosystem conventions: `tempfile` for sandboxed directories, `assert_cmd` for binary invocation, `insta` for snapshotting stable CLI output where helpful, and hand-rolled recording mocks rather than `mockall` because the assertions of interest are about ordered call sequences.

## Out of Scope

- Removal commands (`project rm`, `project branch rm`) and explicit container lifecycle commands (`stop`, `start`, `image rebuild`).
- A reconciliation command (`project doctor`) for cleaning up drift between the registry and reality.
- Concurrency safety. Single-user, single-driver assumption holds for MVP; no file locks, no parallel-build coordination. SQLite WAL mode is enabled defensively but no other locking is added.
- Alternative container backends (Docker, Podman). The trait exists; only the Apple `container` implementation ships.
- A TUI or graphical session browser.
- Linux or x86_64 macOS support.
- Auto-attaching at the end of `branch`. The two-step `branch` then `attach` flow is the MVP.
- A prompt or confirmation step when `flake.lock` changes; the MVP rebuilds silently.
- Multiple sessions per branch container (e.g. a plain shell alongside the Claude session). The architecture allows this trivially later, but the MVP exposes only the Claude session.

## Further Notes

- The single-bind-mount worktree model depends on git's `worktree.useRelativePaths` config being applied to the bare repository at clone time. Without it, the `.git` pointer file inside each worktree contains an absolute host path that does not resolve inside the container.
- The base image is intentionally small: Nix plus the `claude-code` package and a minimal userland. tmux is deliberately not installed in the image because the host-side session model means the container never runs it.
- The image rebuild policy is keyed on `flake.lock` hash, not `flake.nix` content. Editing `flake.nix` without committing a `flake.lock` change does not trigger a rebuild; the container's writable layer absorbs any incremental Nix downloads. This is the right tradeoff because `flake.lock` is the closed-over input that determines the realized store closure.
- The trait-based backend abstraction is the single most important design lever for agent-driven development of this codebase: it means every change to orchestration logic is unit-testable in milliseconds against recording mocks, and end-to-end tests exist solely to validate that the real CLI wrappers correctly produce the invocations the unit tests assert on.
